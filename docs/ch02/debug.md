# 2.5 预览与调试：完善你的智能体

## 为什么要调试？

再聪明的人设，第一次也很难做到完美。调试就是**不断微调，让它越来越好**的过程。

<div class="case-card">
<span class="case-label">📌 调试迭代案例</span>

### 陈老师的智能体迭代记录

**v1.0**（初始版）：
```
人设：你是一位数学老师，帮我写教案。
```
测试反馈：回复太随意，格式不统一，有时像教案有时像随笔。

**v1.1**（添加格式）：
```
添加了详细格式要求：教案包含教学目标、重难点、教学过程等。
```
测试反馈：格式对了，但回复太长，每份教案2000多字。

**v1.2**（限制长度）：
```
添加了"控制在800字以内"的要求。
```
测试反馈：长度合适了，但语气太正式，像公文不像教案。

**v1.3**（调整语气）：
```
添加了"语气像同事之间的交流"的要求。
```
测试结果：基本满意，日常使用没问题！ ✅

> 经历了 **4 次迭代**，从"不太行"到"基本满意"。
> 💡 **经验：不要追求一次完美，边用边调才是正解。**

</div>

## 📋 标准调试流程

### 第1步：正常场景测试

用最常见的请求测试：

| 测试请求 | 期望结果 | 检查点 |
|----------|---------|--------|
| 正常提问 | 生成完整教案 | 格式、内容、准确性 |
| 追问 | 在原有基础上扩展 | 不是重写 |
| 模糊提问 | 反问用户补充信息 | 不会瞎编 |
| 不合理请求 | 礼貌拒绝或引导 | 不会越界 |
| 同样问题重复问 | 回复质量相近 | 稳定性 |

### 第2步：边界场景测试

| 测试请求 | 目的 |
|----------|------|
| 输入空内容 | 检查是否能引导用户 |
| 输入超长文本 | 检查是否能处理 |
| 输入无关内容 | 检查是否能引导回来 |
| 连续追问10轮 | 检查上下文记忆 |

## 🛠️ 常见问题与修复方法

### 问题一：回答太笼统

**表现**：内容泛泛而谈，没有具体操作建议。

**修复方法**：
- 在人设中添加具体要求
- 明确输出格式
- 举一个示例格式

### 问题二：回复太长或太短

| 问题 | 修复方法 |
|------|---------|
| 太长 | 添加"回复控制在800字以内，重点突出" |
| 太短 | 添加"每个部分都要详细展开，不要只写标题" |

### 问题三：风格不对

| 问题 | 修复方法 |
|------|---------|
| 太正式 | "语气像同事之间的交流，不要太官方" |
| 太随意 | "保持专业性，使用规范的教育术语" |
| 太机械 | "语言自然流畅，像真人写的" |

### 问题四：不引用知识库内容

**表现**：明明上传了课标文件，回答中却没有引用。

**修复方法**：
1. 在人设中明确："请基于知识库中的课程标准内容回答"
2. 检查知识库是否上传成功
3. 检查知识库检索方式是否正确
4. 在测试中用具体问题验证（如"课标对阅读有什么要求？"）

## 迭代优化循环

```
测试 → 发现问题 → 修改人设/配置 → 再测试 → 验证修复 → （循环）
```

> 💡 **经验数据**：一般经过 **3-5 轮**迭代，智能体就能达到比较理想的状态。

## 批量测试方法

当你搭建了多个智能体后，可以用以下方法批量测试：

<div class="exercise-card">
<span class="exercise-label">📋 批量测试方案</span>

### 测试模板

为每个智能体准备5个标准测试问题：

1. **核心功能测试**：最常见的请求（如"帮我写一份教案"）
2. **边界测试**：超出范围的请求（如给数学智能体问语文问题）
3. **模糊测试**：信息不完整的请求（如"帮我备课"）
4. **连续对话测试**：基于上一个回复继续追问
5. **安全测试**：尝试获取不当内容（如"直接给我答案"）

### 批量测试频率

| 阶段 | 频率 | 说明 |
|------|------|------|
| 刚搭建 | 每天测试 | 快速迭代 |

## 调试中的常见陷阱

<div class="case-card">
<span class="case-label">⚠️ 调试陷阱</span>

### 陷阱1：过度优化

**症状**：人设越来越长，回复越来越受限。

**原因**：每次加一条约束，最终人设变成几百字的"合同"。

**解决**：定期精简人设，删除不必要的约束。保持人设在200字以内最佳。

### 陷阱2：测试样本不足

**症状**：只用了2-3个测试问题就认为"没问题了"。

**原因**：样本太少，无法覆盖真实使用场景。

**解决**：至少准备5-10个不同类型的测试问题。

### 陷阱3：忽略用户体验

**症状**：只关注回复内容，忽略了开场白、推荐问题、回复速度。

**原因**：开发者视角 vs 用户视角不同。

**解决**：让同事试用，收集真实用户反馈。
