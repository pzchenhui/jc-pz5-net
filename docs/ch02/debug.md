# 2.5 预览与调试：完善你的智能体

## 为什么要调试？

再聪明的人设，第一次也很难做到完美。调试就是**不断微调，让它越来越好**的过程。

<div class="case-card">
<span class="case-label">📌 调试迭代案例</span>

### 陈老师的智能体迭代记录

**v1.0**（初始版）：
```
人设：你是一位数学老师，帮我写教案。
```
测试反馈：回复太随意，格式不统一，有时像教案有时像随笔。

**v1.1**（添加格式）：
```
添加了详细格式要求：教案包含教学目标、重难点、教学过程等。
```
测试反馈：格式对了，但回复太长，每份教案2000多字。

**v1.2**（限制长度）：
```
添加了"控制在800字以内"的要求。
```
测试反馈：长度合适了，但语气太正式，像公文不像教案。

**v1.3**（调整语气）：
```
添加了"语气像同事之间的交流"的要求。
```
测试结果：基本满意，日常使用没问题！ ✅

> 经历了 **4 次迭代**，从"不太行"到"基本满意"。
> 💡 **经验：不要追求一次完美，边用边调才是正解。**

## 📋 标准调试流程

### 第1步：正常场景测试

用最常见的请求测试：

| 测试请求 | 期望结果 | 检查点 |
|----------|---------|--------|
| 正常提问 | 生成完整教案 | 格式、内容、准确性 |
| 追问 | 在原有基础上扩展 | 不是重写 |
| 模糊提问 | 反问用户补充信息 | 不会瞎编 |
| 不合理请求 | 礼貌拒绝或引导 | 不会越界 |
| 同样问题重复问 | 回复质量相近 | 稳定性 |

### 第2步：边界场景测试

| 测试请求 | 目的 |
|----------|------|
| 输入空内容 | 检查是否能引导用户 |
| 输入超长文本 | 检查是否能处理 |
| 输入无关内容 | 检查是否能引导回来 |
| 连续追问10轮 | 检查上下文记忆 |

## 🛠️ 常见问题与修复方法

### 问题一：回答太笼统

**表现**：内容泛泛而谈，没有具体操作建议。

**修复方法**：
- 在人设中添加具体要求
- 明确输出格式
- 举一个示例格式

### 问题二：回复太长或太短

| 问题 | 修复方法 |
|------|---------|
| 太长 | 添加"回复控制在800字以内，重点突出" |
| 太短 | 添加"每个部分都要详细展开，不要只写标题" |

### 问题三：风格不对

| 问题 | 修复方法 |
|------|---------|
| 太正式 | "语气像同事之间的交流，不要太官方" |
| 太随意 | "保持专业性，使用规范的教育术语" |
| 太机械 | "语言自然流畅，像真人写的" |

### 问题四：不引用知识库内容

**表现**：明明上传了课标文件，回答中却没有引用。

**修复方法**：
1. 在人设中明确："请基于知识库中的课程标准内容回答"
2. 检查知识库是否上传成功
3. 检查知识库检索方式是否正确
4. 在测试中用具体问题验证（如"课标对阅读有什么要求？"）

| 5.23 | v1.3 | 调整语气 | ✅ 基本满意 | 微调细节 |

## 迭代优化循环

```
测试 → 发现问题 → 修改人设/配置 → 再测试 → 验证修复 → （循环）
```

> 💡 **经验数据**：一般经过 **3-5 轮**迭代，智能体就能达到比较理想的状态。
